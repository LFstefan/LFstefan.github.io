线程A等线程B执行完了之后再执行

  • thread.join()方法：调用线程等待执行此方法的的线程

为什么会存在并发

  1. 竞态条件
    check-then-act（例如：if判定条件）
    read-modify-write（例如：i++自增由三个动作完成：读取，修改，写入）
  2. 数据资源竞争
    各个线程的访问顺序不同可能会造成不同的结果
  3. cpu和主内存之间的高度缓存：缓存一致协议

缓存一致协议

  • 现代CPU都是多核心+多级缓存架构，如果我们想要保证工作在不同核心上的线程读取到的数据都是一致的，最简单的做法是保证所有读写操作直接在内存上发生；如果真的这么做，会导致L1/L2/L3三级缓存全部失效，而且所有读写操作全部都会落在内存总线上，效率极其低下。

并发的解决方案

  • 同步=互斥（互斥锁）：保证多个线程无法同时访问同一临界区资源，只能串行访问

    • 同步通过监听器来实现，每一个Java对象和一个监听器来关联，线程的加多锁和解锁操作对象是监听器，而不是对象本身
    • 锁其实就是一个标识

  • synchronized（强同步：同时满足互斥性和可见性）

    • 同步方法：public int…，锁住的是持有该方法的整个对象
    • 同步块：synchronized（被锁对象实例）{…}

何为死锁

  • 死锁：两个线程互相等待对方持有的资源，A持有c，等待d；B持有d，等待c

如何避免死锁的发生（因为Java和jvm都没有提供避免死锁的方式，所以只有程序来避免）

  • 避免同步块调用其他的同步快
  • 活锁：线程持续重试一个总是失败的操作
  • 饿死：总有线程可以抢占他的资源，导致该线程一致无法执行

volatile：一种较弱的同步，只满足可见性
确保每条线程只会从主内存中读取该变量的值，而不会从cpu自身的高度缓存中去读，从而保证了可见性
final关键字本身就具有安全访问，无需同步的特性（不可变类不会涉及到多线程的问题）
final和volatile不能同时存在
volatile实现原理

- 通过内存屏障插入指令之间的策略来解决数据的可见性问题
- 每个volatile变量写之前--->插入storestore屏障
- 每个volatile变量写之后--->插入storeload屏障
- 每个volatile变量读之后--->插入loadload屏障和loadstore屏障

JVM内存模型

  • 并发中，线程如何通信，以及如何同步数据

    • 命令式编程中，通信方式有两种
      • 共享内存：读写内存中公共状态
      • 消息传递：发送消息
    • 同步其实指的是线程操作发生的相对顺序

  • java采用的是共享内存

    • 所有的静态定义，实例，数组均存放在堆内存中，所有线程共享
    • 局部变量，方法参数，异常处理器参数；均存放在每个线程栈中，各个线程不共享，不会存在内存可见性的问题

  • jvm决定了java线程之间的隔离性

    • 就是说线程A对共享变量的更改何时对线程B可见

  • 相对于主内存而言，线程的本地内存就是指CPU的高速缓存和寄存器等每个CPU独有的存储介质

  • CPU写缓冲区的好处

    • 保证指令流水线持续执行
    • 避免cpu等待内存的写入，提高CPU执行性能
    • 可以批处理刷新缓冲区，将数据批量写入主内存
    • 每个cpu拥有各自的写缓冲区，互不可见

  • 指令重排

    • 编译器的指令重排：不会影响单线程程序语义
    • 指令级别的并行重排，指令并行执行
    • 内存系统的重排序

  • 内存屏障：用来冲刷高速缓冲区，禁止指令重排，保证内存可见性的

    • loadload
    • storeload
    • loadstore
    • storestore
    • load指的是加载读取数据，store指的是写数据
    • 用在指令与指令执行之间

  • 数据依赖性：指令重排会对结果造成影响

    • 编译器和处理器的指令重排都对单个CPU中有数据依赖性的操作不会发生指令重排，但是多核CPU中情况就不适用了

  • 线程中断
    Thread currentThread = Thread.currentThread();
    currentThread.interrupt();//中断当前线程
    Thread.interrupted();//判定当前线程的中断状态，同时会清除中断标识位
    currentThread.isInterrupted();//判定调用线程的中断状态，不会清除中断标识位

publicvoidreleaseConnection(Connectionconnection){
if(connection!=null){

pool.addLast(connection);
pool.notifyAll();
}

手写线程池

客户端和服务器端的交互
http请求—实质都是socket—TCP请求
客户端线程池—发送请求和接受请求分离
发送请求线程，发送请求完成后便直接返回处理其他事情
专门的接受请求线程负责接受请求响应
服务器端—同样接受请求和处理请求分离
接受请求线程接受请求后封装成统一格式发给处理线程便返回继续处理其他请求
处理请求线程负责接受统一格式的封装请求做处理，然后交给相应线程返回信息

所有的过程全部采用异步的方式进行，实现性能的提升
但是需要注意的是，如何标记请求和相应之间的对应关系
